# VeriVote 项目计划

## 项目名称

VeriVote：基于零知识证明与承诺聚合的隐私保护可信投票审计平台。

## 项目定位

VeriVote 面向信息安全竞赛场景，目标是构建一个可演示、可验证、可扩展的隐私保护投票审计平台。系统不直接公开用户的明文投票内容，而是通过承诺机制、Merkle 树、聚合审计和后续零知识证明能力，实现“投票隐私可保护、投票过程可审计、异常行为可演示”的综合安全方案。

项目首先以 MVP 形式完成核心流程闭环，包括用户注册、投票、回执查询、选票聚合、公告板公开、验证端审计和攻击演示。后续逐步加入重复投票检测、非法投票检测、链上存证、Solidity 智能合约和简化 ZK 证明。

## 核心目标

1. 用户可以注册、投票、查看投票回执。
2. 系统不直接公开明文投票，而公开 commitment。
3. 聚合器收集选票，检查重复投票，生成聚合结果。
4. 公告板公开 commitment、Merkle Root 和聚合结果。
5. 验证端可以验证选票未被篡改、未重复投票、总票数一致。
6. 演示端可以模拟篡改、重复投票、非法投票和伪造结果。
7. 后期加入 Solidity 智能合约和简化 ZK 证明。

## 主要角色

| 角色 | 职责 |
| --- | --- |
| 投票用户 | 注册身份、提交投票、保存回执、查询自己的选票是否被纳入公告板 |
| 管理员 | 创建投票活动、配置候选项、启动或关闭投票、查看审计概览 |
| 聚合器 | 收集选票承诺、检测重复投票、计算聚合结果、生成 Merkle Root |
| 公告板 | 公开 commitment、Merkle Root、聚合结果和审计元数据 |
| 验证者 | 独立验证回执、Merkle inclusion、总票数和聚合一致性 |
| 攻击演示者 | 触发篡改、重复投票、非法投票、伪造结果等演示场景 |
| 链上审计合约 | 后期用于存证 Merkle Root、聚合结果摘要和审计事件 |

## 功能范围

### MVP 阶段功能

1. 用户注册与身份标识生成。
2. 用户提交投票并生成本地回执。
3. 系统生成并保存投票 commitment。
4. 聚合器读取投票记录并生成聚合结果。
5. 公告板展示 commitment、Merkle Root 和聚合结果。
6. 验证端根据回执验证选票是否被纳入。
7. 攻击演示端模拟基础异常并展示验证失败。

### 暂不纳入 MVP 的功能

1. 真实生产级身份认证。
2. 完整密码学级别的零知识证明电路。
3. 主网或测试网部署。
4. 大规模并发投票能力。
5. 复杂权限系统和组织管理系统。

## 安全设计重点

1. 明文投票与公开数据分离。
2. 使用 commitment 降低投票内容泄露风险。
3. 使用 Merkle Root 固化投票集合状态。
4. 使用 nullifier 或唯一投票标识检测重复投票。
5. 使用公告板提供公开、可复验的数据源。
6. 使用攻击演示端展示异常行为如何被发现。
7. 后期使用链上存证降低公告板被事后篡改的风险。

## 数据对象概览

| 对象 | 说明 |
| --- | --- |
| voterId | 用户身份标识，MVP 中可使用模拟身份 |
| vote | 明文投票内容，仅在受控流程中使用，不直接公开 |
| salt | commitment 随机盐，用于隐藏投票内容 |
| commitment | 对投票内容和盐的承诺值，公开在公告板 |
| nullifier | 用于检测重复投票的唯一标识 |
| receipt | 用户回执，包含 commitment、索引、Merkle proof 等验证材料 |
| merkleRoot | 所有合法投票 commitment 构成的 Merkle 树根 |
| tally | 聚合统计结果 |
| auditLog | 审计事件记录，包括异常检测结果和关键状态变化 |

## 成功标准

1. 能完整演示一次投票流程：注册、投票、回执、公告、验证。
2. 公告板中不暴露明文投票。
3. 验证端能够发现至少四类异常：篡改、重复投票、非法投票、伪造结果。
4. 聚合结果与公告板 commitment 集合保持一致。
5. 后续扩展到链上存证和简化 ZK 证明时，不需要推翻 MVP 架构。
