# VeriVote 项目现状说明与后续计划

> 说明：本文档记录当前分支已经完成了什么、项目现在处于什么状态、还缺哪些关键能力，以及下一步应该按什么顺序完善。  
> 当前分支重点是把后端迁移到 Python / FastAPI，并继续沿着 Haechi、Aggios、Zeeperio 三篇论文的思路，把项目补成一个能演示、能解释、能验证的比赛原型。

## 0. 完成版状态（2026-05-19）

当前项目已经按本文档的七阶段计划完成到“比赛原型可交付”状态。项目现在具备：

1. **主闭环**：创建选举、添加候选人、注册用户、投票、生成 receipt、公告板、聚合器、结果验证。
2. **Haechi 风格挑战审计**：pending ballot、cast-or-challenge、opening 验证、challenge 不进入正式 tally。
3. **Aggios 风格聚合器**：重复票检测、非法票检测、有效票 tally、audit report、tally consistency check。
4. **Zeeperio 风格公开审计**：Merkle Root、commitmentRoot、receiptRoot、auditHash、tallyHash、local mock chain audit、artifact bundle 导出。
5. **系统安全测试**：`scripts/api_smoke_test.py` 已经把主闭环和安全矩阵变成可执行验收脚本。
6. **ZK / Pedersen 实验模块**：Mock / Real ZK validity proof 接口、tally correctness proof 骨架、Pedersen opening 和 aggregate verify。
7. **比赛材料**：`docs/DEMO_SCRIPT.md`、`docs/COMPETITION_DELIVERY.md`、`docs/SECURITY_TESTS.md` 已经能支撑演示和答辩。

完整验收命令：

```bash
pnpm verify:plan
```

如果只验证后端闭环与安全矩阵：

```bash
python scripts/api_smoke_test.py
```

因此，后续不再是补 MVP 缺口，而是根据比赛时间继续做展示优化、录屏、PPT、真实链上部署或更严格的论文级证明增强。

## 4. 当前还没有完全做完的部分

下面这些部分是目前最关键的“差距”，也是后续必须继续补的地方。

### 4.1 加密承诺还不是论文原版

当前承诺机制还是工程化简化版，核心仍偏向：

```text
commitment = sha256(electionId + voteVector + randomness)
```

这和论文里的 Pedersen / 向量承诺还不是同一个层级。

也就是说：

1. 现在能演示“承诺”。
2. 但还不能说已经复现了论文级别的密码学构造。
3. 后续如果要更接近论文，需要做 Pedersen 或更正式的向量承诺实验模块。

### 4.2 回执机制还不是完整的 confirmation-code chain

当前回执更偏向查询和验证用途，还没有完整做到论文里那种链式确认码。

也就是说：

1. 现在能做“我收到了票”的验证。
2. 但还没有完全做成一条可以检查删除、重排、篡改的回执链。
3. 后续需要把 receipt 变成一个更强的链式审计机制。

### 4.3 聚合器还没有达到论文级的 EPA

当前的 aggregator 更像工程化聚合器，能完成基本的收集、去重、统计、输出审计结果，但还没有到 Aggios 论文里那种完整的 extended partition argument 级别。

这意味着：

1. 现在能做批量计票。
2. 能做重复投票检测。
3. 能出 audit report。
4. 但还没有严格的聚合证明体系。

### 4.4 链上审计还停留在摘要级

当前项目对 Zeeperio 的吸收，主要还是“摘要留痕”和“可验证审计”的方向，还没有做到完整的 Solidity / zk-SNARK 验证流程。

现在能做的更多是：

1. 保存摘要。
2. 保存 Merkle Root / tallyHash / proofHash 一类信息。
3. 让验证端能交叉核对。

但还没有做到：

1. 完整链上验证。
2. 完整 ZK proof 验证。
3. 全流程自动审计合约。

### 4.5 安全测试还不够系统

虽然项目已经有攻击演示思路，但还需要更系统的测试矩阵，尤其是：

1. 重复投票。
2. 非法候选项。
3. 篡改 commitment。
4. 删除或替换公告板数据。
5. 伪造结果。
6. 伪造 proof。
7. 服务重启后的状态恢复。

现在更多是“知道要测什么”，还没有完全变成统一、自动化、可复用的测试集合。

## 5. 下一步应该怎么做

下面是建议的后续开发顺序。这个顺序不是随便排的，而是按“先把主闭环跑稳，再加审计，再加安全，再加论文加分项”的思路来的。

### 5.1 第一阶段：把主闭环彻底稳定

#### 目标

把“创建选举 -> 用户投票 -> 生成回执 -> 公告板公开 -> 聚合计票 -> 验证结果”这条主链路完全跑稳。

#### 要做的事

1. 检查前端到后端的接口是否完全对齐。
2. 确认投票流程没有断点。
3. 确认公告板展示的数据一致。
4. 确认验证端能读到正确的摘要。
5. 确认后端服务重启后不会把基本状态弄乱。

#### 产出

1. 一个稳定的 MVP。
2. 一条可重复演示的投票链路。
3. 一套能给评委看的最小闭环。

#### 验收标准

1. 用户能完整投一次票。
2. 能查到回执。
3. 公告板能展示结果摘要。
4. 验证端能解释结果正确或失败的原因。

### 5.2 第二阶段：补 Haechi 风格的挑战审计

#### 目标

引入 `cast-or-challenge` 逻辑，让系统不仅能“投票”，还可以“公开挑战并验证记录”。

#### 要做的事

1. 增加 challenge 模式。
2. 增加 pending ballot 或挑战记录。
3. 增加公开 opening 或审计信息。
4. 将回执和挑战信息连起来。
5. 明确 challenge 只是审计行为，不是投票反悔功能。

#### 产出

1. 更像 Haechi 的审计流程。
2. 更明确的“隐私 + 可验证”叙事。
3. 更适合演示“票已记录但不泄密”。

#### 验收标准

1. 能成功 challenge 已记录的投票。
2. 能验证 challenge 的 opening。
3. 能证明被 challenge 的内容不会进入正式 tally。

### 5.3 第三阶段：补 Aggios 风格的聚合器

#### 目标

把“单票收集”推进到“批量聚合和重复检测”，让系统能解释为什么高频投票场景下仍然可控。

#### 要做的事

1. 加强 `voteTokenHash` 或等价标识。
2. 强化重复投票检测。
3. 让 aggregator 明确区分有效票、无效票和重复票。
4. 输出 batch tally 和 audit report。
5. 让公告板可以直接展示聚合结果摘要。

#### 产出

1. 一个更成熟的 aggregator service。
2. 一个可解释的批处理投票结果。
3. 一个可以讲清楚“为什么没算某张票”的系统。

#### 验收标准

1. 重复票能被稳定识别。
2. 非法票不会进入正式结果。
3. 聚合结果和公告板结果一致。

### 5.4 第四阶段：补 Zeeperio 风格的公开审计

#### 目标

让项目具备“公开留痕、可复核、可上链摘要”的能力。

#### 要做的事

1. 统一 `proofHash`、`tallyHash`、`Merkle Root` 这些摘要字段。
2. 让审计结果可以被导出。
3. 如果需要，补一个简化的 Solidity 合约骨架。
4. 让验证端能交叉比对链上摘要和链下公告板。

#### 产出

1. 更像 Zeeperio 的公开审计层。
2. 更适合答辩的审计证据链。
3. 更明确的链上 / 链下分工。

#### 验收标准

1. 关键摘要可导出。
2. 审计信息可复核。
3. 结果可解释，不是黑盒。

### 5.5 第五阶段：系统化安全测试

#### 目标

把“知道哪些攻击会出问题”变成“有一套统一测试矩阵可以覆盖这些问题”。

#### 建议测试项

1. 篡改 commitment。
2. 删除 ballot。
3. 重放 vote 请求。
4. 重复投票。
5. 非法候选项。
6. 伪造 receipt。
7. 伪造 tally。
8. 伪造 proof。
9. 替换公告板摘要。
10. 服务重启后的状态恢复。

#### 产出

1. `docs/SECURITY_TESTS.md` 变成真正可执行的测试矩阵。
2. 后端和前端都有对应的攻击演示。
3. 答辩时可以直接展示系统防护边界。

#### 验收标准

1. 每类攻击都有预期和实际结果。
2. 能明确说出攻击被谁发现、在哪里发现、为什么失败。

### 5.6 第六阶段：ZK / Pedersen 实验模块

#### 目标

增加一个实验性质的密码学增强模块，用来展示更接近论文原版的构造思路，但不替换当前主流程。

#### 要做的事

1. 增加 Pedersen 风格承诺实验。
2. 增加向量承诺或聚合承诺演示。
3. 增加一个简化的 ZK 证明演示。
4. 明确标注该模块是实验模块。

#### 产出

1. 一个高阶加分项。
2. 一个可以讲密码学故事的模块。
3. 一个和主流程解耦的研究展示层。

#### 验收标准

1. 能展示成功 / 失败。
2. 不影响现有投票主流程。
3. 答辩中能清楚解释其安全意义。

### 5.7 第七阶段：整理比赛材料

#### 目标

把代码成果转成评委能快速理解的材料。

#### 要做的事

1. 整理项目背景。
2. 整理系统架构图。
3. 整理数据流图。
4. 整理威胁模型图。
5. 准备演示脚本。
6. 准备测试案例截图。
7. 准备答辩 PPT。
8. 准备录屏和讲解稿。

#### 产出

1. 可以直接展示的 PPT。
2. 可以直接演示的录屏。
3. 可以直接交付的项目说明书。

#### 验收标准

1. 评委不需要看代码就能理解项目。
2. 评委能理解项目的创新点和防护价值。

## 7. 当前项目的定位

这件事要一直讲清楚：

1. 当前项目是一个比赛原型，不是生产级电子投票系统。
2. 当前项目的目标是“能演示、能验证、能解释”。
3. 当前项目不是三篇论文的逐字复现，而是工程化吸收它们的核心思想。
4. 当前项目最重要的是把“隐私、可验证、抗攻击、可审计”这四件事做成一条完整链路。
