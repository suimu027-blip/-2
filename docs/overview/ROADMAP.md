# VeriVote 路线图

## 阶段一：MVP

目标：完成隐私保护可信投票审计平台的最小闭环，让项目可以被完整演示。

核心任务：

1. 设计用户注册、投票、回执查看流程。
2. 设计 commitment 生成规则。
3. 设计聚合器收集选票和检测重复投票的流程。
4. 设计公告板公开 commitment、Merkle Root 和聚合结果的格式。
5. 设计验证端验证回执、Merkle proof 和总票数一致性的流程。
6. 设计攻击演示端的基础异常场景。

验收标准：

1. 用户可以完成注册、投票和查看回执。
2. 公告板不公开明文投票。
3. 聚合器能生成 Merkle Root 和聚合结果。
4. 验证端能验证正常投票。
5. 攻击演示端能展示篡改、重复投票、非法投票、伪造结果。

## 阶段二：安全增强

目标：提升系统对异常输入、内部篡改和审计争议的抵抗能力。

核心任务：

1. 引入投票活动编号，避免跨活动重放。
2. 强化 nullifier 设计，提升重复投票检测能力。
3. 增加输入合法性校验，包括候选项范围、投票状态和请求格式。
4. 增加审计日志，记录注册、投票、聚合、公告和验证事件。
5. 为审计日志设计哈希链或摘要机制。
6. 完善验证端的异常解释能力。

验收标准：

1. 重复投票可以被稳定发现。
2. 非法投票不会进入合法聚合结果。
3. 审计日志能追踪关键操作。
4. 验证端能给出清晰的失败原因。
5. 攻击演示端能复现并解释更多异常场景。

## 阶段三：链上审计

目标：使用 Solidity 智能合约对关键摘要进行存证，降低公告板和聚合结果被事后替换的风险。

核心任务：

1. 设计链上审计合约的数据结构。
2. 将投票活动编号、Merkle Root 和聚合结果摘要写入合约。
3. 记录公告批次和时间戳。
4. 设计链上读取接口，供验证端交叉验证。
5. 设计链下公告板与链上摘要的一致性检查流程。

验收标准：

1. Merkle Root 可以被上链存证。
2. 聚合结果摘要可以被上链存证。
3. 验证端可以对比链上数据和公告板数据。
4. 公告板被替换时，验证端能够发现 root 不一致。
5. 链上只存储摘要，不泄露明文投票。

## 阶段四：ZK 增强

目标：加入简化零知识证明能力，增强项目的密码学创新性和比赛展示价值。

核心任务：

1. 设计简化 ZK 证明目标。
2. 证明投票属于合法候选项集合，但不公开具体选择。
3. 证明用户拥有合法投票资格，但不公开完整身份。
4. 探索聚合结果可验证证明。
5. 将 ZK 验证结果接入验证端展示。
6. 准备适合答辩讲解的 ZK 流程图和演示样例。

验收标准：

1. 能展示至少一个简化 ZK 证明流程。
2. 验证端能展示 ZK 验证通过或失败。
3. ZK 增强不破坏原有 MVP 流程。
4. 答辩中可以清楚解释证明的输入、输出和安全意义。

## 阶段五：比赛材料整理

目标：将项目工程成果转化为完整的比赛展示材料。

核心任务：

1. 整理项目背景、痛点和创新点。
2. 制作系统架构图和数据流图。
3. 编写演示脚本，包括正常流程和攻击流程。
4. 准备测试用例和验证截图。
5. 总结安全设计、威胁模型和防护效果。
6. 准备答辩 PPT、项目说明书和录屏材料。

验收标准：

1. 能在限定时间内完整演示项目。
2. 正常流程和攻击流程都能稳定运行。
3. 文档能支撑评委理解项目架构与安全价值。
4. 比赛材料突出隐私保护、可验证审计、链上存证和 ZK 增强。
5. 项目后续扩展方向清晰。
