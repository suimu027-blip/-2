# VeriVote 威胁模型

## 安全目标

VeriVote 的安全目标是让投票过程在保护隐私的同时具备可审计性。系统需要避免明文投票泄露、选票被篡改、重复投票、非法投票、聚合结果伪造和公告板事后替换等问题。

## 保护资产

| 资产 | 说明 | 安全需求 |
| --- | --- | --- |
| 明文投票 vote | 用户真实选择 | 保密性 |
| salt | commitment 随机盐 | 保密性 |
| commitment | 投票承诺值 | 完整性、可公开验证 |
| nullifier | 重复投票检测标识 | 完整性、适度隐私 |
| receipt | 用户投票回执 | 完整性、可验证性 |
| Merkle Root | 投票集合摘要 | 完整性、不可抵赖性 |
| 聚合结果 tally | 统计结果 | 完整性、可复验性 |
| 审计日志 auditLog | 异常和状态变化记录 | 完整性、可追踪性 |
| 链上摘要 | 后期存证数据 | 完整性、时间顺序可信 |

## 信任边界

1. 用户端与聚合器之间：用户提交的数据可能被伪造、重放或篡改。
2. 聚合器与公告板之间：聚合结果和 commitment 列表可能被替换。
3. 公告板与验证端之间：公告板展示的数据可能不完整或不一致。
4. 攻击演示端与正式流程之间：攻击数据不得污染正常投票数据。
5. 链下系统与链上审计端之间：链下摘要写入链上前可能被恶意替换。

## 攻击者模型

### 普通恶意用户

能力：

1. 重复提交投票。
2. 修改本地请求参数。
3. 提交非法候选项。
4. 尝试伪造回执。

目标：

1. 让自己多投票。
2. 投给不存在的候选项。
3. 声称自己的票被篡改。

### 恶意管理员或内部人员

能力：

1. 修改聚合结果。
2. 删除部分 commitment。
3. 替换公告板数据。
4. 隐藏异常日志。

目标：

1. 操纵最终统计结果。
2. 让特定用户选票消失。
3. 事后修改投票集合。

### 外部攻击者

能力：

1. 抓包重放请求。
2. 尝试枚举投票内容。
3. 尝试构造碰撞或伪造 commitment。
4. 尝试干扰公告板数据展示。

目标：

1. 泄露用户投票隐私。
2. 破坏系统完整性。
3. 造成验证端误判。

### 演示攻击者

能力：

1. 在攻击演示端触发预设异常场景。
2. 构造篡改数据、重复数据、非法数据和伪造统计。

目标：

1. 展示系统安全能力。
2. 证明验证端可以发现异常。

## 主要威胁与缓解措施

| 威胁 | 描述 | 影响 | 缓解措施 |
| --- | --- | --- | --- |
| 明文投票泄露 | 公告板或日志暴露 vote | 用户隐私受损 | 公告板只公开 commitment；日志避免记录明文 vote |
| salt 泄露 | salt 与 commitment 同时公开导致可枚举投票 | 隐私降低 | salt 只进入用户回执或受控验证流程，不在公告板公开 |
| 重复投票 | 同一用户提交多张有效票 | 结果失真 | 使用 nullifier 检测重复；公告重复检测摘要 |
| 非法投票 | 投给不存在候选项 | 结果污染 | 聚合器校验候选项范围；验证端复查总票数 |
| commitment 被篡改 | 公告板替换某条 commitment | 用户选票丢失或被替换 | Merkle proof 验证；用户回执对照公告板 |
| Merkle Root 被替换 | 公告板整体替换投票集合 | 审计失效 | 后期将 Merkle Root 上链存证 |
| 聚合结果伪造 | tally 与真实 commitment 集合不一致 | 选举结果失真 | 验证端检查总票数一致性；后期加入可验证聚合证明 |
| 回执伪造 | 用户或攻击者构造假 receipt | 争议增加 | receipt 绑定 commitment、索引、活动编号和 Merkle proof |
| 重放攻击 | 复制旧投票请求再次提交 | 重复投票 | nullifier 去重；投票活动编号绑定 |
| 内部篡改审计日志 | 管理员删除异常记录 | 事后追责困难 | 审计日志只追加；后期关键摘要上链 |

## MVP 重点防护

1. 不公开明文投票。
2. 对每张投票生成 commitment。
3. 使用 nullifier 检测重复投票。
4. 使用 Merkle Root 固定合法投票集合。
5. 使用 receipt 和 Merkle proof 让用户验证自己的票已被纳入。
6. 验证聚合结果总票数是否与合法 commitment 数量一致。
7. 攻击演示端覆盖篡改、重复投票、非法投票和伪造结果。

## 后续增强方向

### 安全增强

1. 增加用户签名，证明投票请求来自合法用户。
2. 对 auditLog 做哈希链，降低日志被删除或重排的风险。
3. 引入更严格的输入校验和状态机。
4. 增加投票活动编号，避免跨活动重放。

### 链上审计

1. 将 Merkle Root 写入 Solidity 合约。
2. 将聚合结果摘要写入合约。
3. 在验证端对比链上 root 与公告板 root。
4. 记录公告批次，形成可追溯的时间线。

### ZK 增强

1. 证明某张票属于合法候选项集合，而不公开具体投票内容。
2. 证明聚合结果由合法 commitment 集合计算得到。
3. 证明用户拥有合法投票资格，而不公开完整身份。
4. 使用简化电路服务比赛演示，优先保证可理解性和可讲解性。

## 残余风险

1. MVP 中的 commitment = Hash(vote, salt) 属于教学型简化方案，不能等同于完整生产级密码协议。
2. 若 salt 生成不安全或重复使用，可能降低投票隐私。
3. 中心化聚合器仍然是重要信任点，需要通过审计日志、验证端和链上摘要逐步降低信任。
4. 若用户丢失 receipt，将难以自行验证自己的选票是否被纳入。
5. ZK 证明在后期加入前，系统只能提供部分可验证性，不能完整证明所有隐私计算过程。
